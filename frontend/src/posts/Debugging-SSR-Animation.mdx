Anyway, the issue started when I tried to use Framer Motion in one of the components. In that component I used the initial attribute with an `opacity` of `0` so that the element would animate into view on page load.
<br/ >
But the animation never happened.
Framer Motion renders elements at their initial state during SSR. In this case that means the element is rendered with `opacity: 0`. After that, Framer Motion relies on client side effects to animate the element to its final state.
<br/ >
If anything disrupts that handoff between the server render and the client hydration, the element can remain stuck in its initial state. With the stack I’m using here, which is React 19 with Framer Motion v12, that handoff wasn’t behaving the way it should.
<br/ >
On a normal static React application where there is no SSR, the animation works perfectly. On a Pyra SSR application, everything appeared instantly with no animation at all.
<br/ >
At that point I realized that this is not something Pyra itself can directly fix. The behavior comes from a third party library interacting with React’s hydration process.
<br/ >
The solution from the Pyra side is the `<ClientOnly>` component.
<br/ >
`<ClientOnly>` skips server side rendering entirely for its children. The server returns `null` or a `fallback`, and the component mounts only after the client is fully ready. When Framer Motion initializes in that environment, it starts fresh with no hydration concerns.
```tsx
import { ClientOnly } from '@pyra-js/adapter-react'
import { motion } from 'framer-motion'

export default function Hero() {
  return (
    <section className="...">
      <ClientOnly fallback={<div className="hero-content-static">...</div>}>
        <motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }}>
          {/* all your framer motion stuff */}
        </motion.div>
      </ClientOnly>
    </section>
  )
}
```
The `fallback` prop allows you to render a static version of the content during SSR so the page is not blank before JavaScript loads. Once the client mounts, the animated version replaces it.
<br/ >
For something like a hero section that is purely visual, you can even skip the fallback entirely and just let the animation appear once the client loads.
<br/ >
After implementing this, I ran into another issue.
<br/ >
This time the error involved the Fast Refresh Babel plugin trying to resolve an import. After digging into the `fast-refresh-plugin.ts` file, I realized what was happening.
<br/ >
This plugin is relatively new, I built it yesterday so It was bound to pop out a few bugs here and e. When the browser bundle includes `@pyra-js/adapter-react` because the user imports `ClientOnly`, esbuild follows the entire static import chain:

```bash
index.js
→ adapter.js
→ fast-refresh-plugin.js
→ @babel/core
```
At that point esbuild tries to bundle @babel/core, which contains an internal reference to:

```typescript
@babel/preset-typescript/package.json
```
That dependency is not installed, so the build fails. Another issue is that this line inside `fast-refresh-plugin.ts`:

```ts
const _require = createRequire(import.meta.url)
```
runs at module load time. That works in `Node.js`, but it crashes in a browser context.
<br/ >
After making some changes, I linked my local Pyra build using npm link and tested it inside my sample project. I immediately got a different error. At this point I realized the fix existed in the source code, but the build output might not have included it, or there might be another esbuild call inside the dev server pipeline that still bundles the problematic code.
<br/ >
The error trace pointed to this line inside a `.cts` handler. It is essentially a static `require()` that esbuild discovers while bundling `@babel/core`.
<br/ >
My fix in `bundler.ts` should have prevented esbuild from entering `@babel/core` entirely by externalizing it, but the error still occurred. That meant there had to be another esbuild call somewhere that did not include the same externalization rule.
<br/ >
Eventually I found the real issue. `compileForServer`, which is used for server side route compilation, only externalizes React. Everything else gets bundled. That includes:

```bash
@pyra-js/adapter-react
→ fast-refresh-plugin.js
→ @babel/core
```
Since this code runs on the server and is loaded with `import()` at runtime where all packages are already available, the correct fix is to externalize all `node_modules` packages.
<br/ >
Once I did that, the error disappeared. So what actually happened here?
<br/ >
When using SSR, the server sends real HTML to the browser. The page is rendered with `renderToString()` and sent as static markup. The browser paints that immediately, which means the content is visible before any JavaScript runs.
<br/ >
Then React hydrates the page. During hydration, React attaches event listeners and takes control of the existing DOM. At this point Framer Motion initializes. However, the motion.div elements already exist in the DOM because they were rendered by the server.
<br/ >
Framer Motion has no way to know whether those elements just appeared or whether they were already there from the beginning. Because of that, it skips the entry animation and jumps straight to the final state.
<br/ >
From Framer Motion’s perspective, there is nothing to animate because the elements already exist.
<br/ >
This behavior is actually intentional. If Framer Motion always respected the initial state during SSR, every page would briefly flash invisible before animating to full opacity. That would look terrible.
<br/ >
So it protects you from that situation, but the side effect is that entry animations silently disappear. The fix for this is Pyra’s `<ClientOnly>` component.
<br/ >
The pattern is similar to approaches used in frameworks like Next.js and Remix.
<br/ >
It works like this:

* On the server, a static fallback renders as normal HTML.
* On the client, after hydration and useEffect, the real component replaces it.

```tsx
<ClientOnly fallback={<div>...static visible content...</div>}>
  <motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }}>
    ...animated content...
  </motion.div>
</ClientOnly>
```
Because the Framer Motion components never appear in the SSR HTML, they are inserted fresh after hydration. Framer Motion then correctly detects them as new elements entering the tree and runs the entry animation.
<br/ >
These fixes will be included in Pyra version `0.27.12`.
